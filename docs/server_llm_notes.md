Server & LLM Notes

- server.js
  - Bootstraps config (default + `config.yaml` + optional `--config-override` yaml + optional loaded-save `gameConfigOverride.yaml`), sets Globals.baseDir/config, and initializes Express + HTTP + RealtimeHub + ModLoader static serving. Maintains in-memory maps for players, things, skills, locations, exits, regions, pending stubs/images, and in-flight generations. Root `defs/*.yaml` files now accept matching mod overlay files from `mods/*/defs/*.yaml`; mod enablement is resolved from merged YAML config `mods.<name>.enabled` first, then `mods/<name>/config.json` `enabled`, with omitted values defaulting to on, and the active mod set is frozen at startup so `/reload_config` reports enable/disable drift but does not hot-toggle running mods. Startup validates active overlays and fails loudly on invalid/unknown definitions, while `reload_config` validates first and preserves the running game if validation fails. AI validation is now backend-aware: `openai_compatible` still requires endpoint/apiKey/model, while `codex_cli_bridge` validates `ai.model` plus `ai.codex_bridge.*`. The `/config` page can also update the loaded game's YAML override live through the same reload path, and save/load persist that raw YAML as `gameConfigOverride.yaml`. NPC single-generation dedupe now keys only on explicit non-empty names so unnamed Add NPC requests can run concurrently. Provides chat utilities (`normalizeChatEntry`, `pushChatEntry`, `scrubGeneratedBrackets`), location/region/thing helpers, save/load wiring (exposed via module.exports), and exposes many of these via `apiScope`.
  - `pushChatEntry` applies bracket/event/think scrub cleanup only to non-user entries; user message content is preserved as entered. New chat entries also persist a `metadata.worldTime` snapshot (`dayIndex`, `timeMinutes`) when world time is initialized so later prompt rendering can label historical entries with their in-world date.
  - Builds `promptEnv`/`viewsEnv` with Nunjucks, loads templates, and defines `prepareBasePromptContext` (builds base prompt context, triggers NPC memory selection jobs). `promptEnv` is rooted at `prompts/`, so includes for shared partials under `prompts/_includes/` must use loader-rooted paths such as `{% include "_includes/travel-prose.njk" %}` rather than bare filenames. Server-side Nunjucks environments include an `eval` filter that renders a string as a nested template against the current context, with optional explicit locals for loop/block-scoped values. Attaches `Globals.getBasePromptContext` and other helpers (e.g., plausibility parsing, attack resolution, status ticking).
  - `parseXMLTemplate(...)` extracts prompt-node content through `Utils.extractXmlNodeContent(...)`: CDATA-bearing prompt nodes are returned as plain text, while ordinary prompt nodes continue to preserve serialized inner XML. `prompts/base-context.xml.njk` wraps its `<generationPrompt>` payload in CDATA so instruction text can mention incomplete/example XML tags without the wrapper XML parser normalizing them into prompt-closing tags.
  - Applied settings can carry `unifiedTonalScale` selections keyed by `defs/unified_tonal_scale.yaml` axes. When all axes are selected, `buildSettingPromptContext(...)` renders a markdown `## Unified Tonal Scale` section with full scale tables plus the selected `THIS STORY` notation/comments; base-context, no-context generic, and slop-remover system prompts insert that section immediately before `config.extra_system_instructions`.
  - Base-context actor need summaries now render as compact `<needs>` bullet lists derived from each active need bar's threshold `sentence`, with `%CHARACTER%` replaced by the actor name; missing sentences are warning-only during live reloads but are validated strictly during actual new-game/load flows so the problem surfaces before a game session starts. `<currentConditions><currentCharacterNeeds>` also mirrors those same sentence lists for the player, party, and on-location NPCs as a quick scene-needs reference block.
  - Base-context `<currentLocation>` now renders only when a non-null `currentLocation` is supplied, which allows generated-region/location follow-up prompts to pass raw `<generatedRegionOrLocation>` XML while intentionally omitting stale location context.
  - `prompts/base-context.xml.njk` now honors root config `prompt_uses_caching`; when that flag is `true`, template-level omissions such as `omitGameHistory` are ignored so `<olderStoryHistory>` remains present for cache-stability experiments. This does not override lower-level `prepareBasePromptContext(...)` exclusion options like `omitCraftHistory` or `omitEventSummaryHistory`.
  - Base-context `<statusEffects>` blocks include each effect's `<name>`, `<description>`, and `<duration>` for locations, characters, inventory, scene items, scenery, and party members.
  - Compact base-context item/inventory summaries include the item's value between level and description, using the setting's currency label, for example `Fractured Sorcerous Orb: Uncommon level 2 (24 gold) - ...`. Full XML mode continues to expose value through each item's `<value>` field.
  - Recent history keeps `event-summary` entries intact by default (scrubber can skip event stripping via `scrub_events`); if their content is empty, `<recentStoryHistory>` renders them from `summaryItems` and strips any event-summary lines containing 🧪. Older and recent history are emitted only through their separate `<olderStoryHistory>` and `<recentStoryHistory>` tags; the server does not append a dangling "Recent story" separator to older history. New event/status summary rows carry exact `summaryItems[].category` metadata for the chat `What changed` drawer; old uncategorized event rows remain valid and display as `Other` client-side. `location-generator-stub` and `region-generator` now opt out of those event-summary history entries when they build base-context so region/location generation is driven by story history instead of mechanical event bundles. Absent-character suffixes union `locationId` and `metadata.traveledToLocationId` when present.
  - When scene summaries are used for older context, hidden entries (`[Hidden from Player]`) and event/status summary-style blocks are skipped from the scene-summary/fallback section so they do not appear inline between summarized scenes.
  - Scene summary parsing accepts `<details>` lines per scene, stores them on `SceneSummaries`, and includes them in older-history scene blocks passed into base-context with compact (single-spaced) bullet rendering. When available, each rendered scene block includes the scene start entry's relative in-world day/time plus calendar date immediately after `Scene N:` (for example `today at 8:12 AM (...)`, `yesterday at 11:59 PM (...)`, or `3 days ago at 8:12 AM (...)`). When overlapping legacy/new scene ranges exist, base-context scene selection prefers the most recently ingested scene coverage per entry index.
  - Scene-summary generation input (used by `/summarize`, `/summarize check`, `/scene_summaries` coverage diagnostics, and scene-threshold summarization) uses `scene_summary_index.js` so every caller shares the same indexed-entry count. The shared index excludes event/status summary-style entries while preserving hidden supplemental/offscreen entries.
  - Scene-summary range validation errors include scalar call diagnostics (`startIndex`, `endIndex`, `redo`, resolved range, and indexed-entry total) without dumping chat history contents.
  - Base prompt context can omit craft/harvest/process history entries when requested (used by crafting/harvest prompts to reduce duplicate actions). Full history entries rendered into base-context prefix non-assistant speakers with their stored in-world timestamp relative to the current in-game day, for example `[Exis][3 days ago at 8:12 AM (Grinday, Thawmance 5, Cycle 1)]`; assistant/Storyteller entries keep the existing single `[Storyteller]` prefix. Older saved entries without a `metadata.worldTime` snapshot are rendered without an invented date label.
- Base prompt context now includes NPC representation summaries (`npcRepresentation`) with race/class counts across all tracked NPCs, and NPC generation prompts include shared guidance to keep race/class variety reasonable without breaking story coherence.
- The shared `prompts/_includes/item.njk` scaffold now emits an XML comment instructing the model to name and describe stacked items as a single instance whenever the seed is missing either the item name or the item description.
- Character inventory generation renders `prompts/_includes/inventory-generator.njk` through `prompts/base-context.xml.njk` with `promptType: "inventory-generator"`, so generated NPC/player inventories receive the same game-state, history, lore, setting, location, region, item, and character context as other base-context prompts while preserving the existing `inventory_generation_*` metadata labels and response parser.
- Prompt-backed item creation, alteration, crafting, and separation now persist the raw prompt-scale attribute bonus list in `thing.metadata.unscaledAttributeBonuses` alongside the scaled applied bonuses; `buildThingPromptItem(...)` prefers that stored raw metadata when present so separation/alter prompts round-trip exact bonus intent instead of reverse-deriving from scaled stats.
- Prompt-backed item alteration treats optional list/effect fields in the returned item XML as authoritative: missing or empty attribute bonuses and `causeStatusEffectOnTarget`/`causeStatusEffectOnEquipper` tags clear stale metadata, while populated tags replace the corresponding scope. Populated target/equipper status effects are synchronously expanded through `StatusEffect.generateFromDescriptions(...)` before persistence/logging, so tool-call alterations and event-driven `alter_item` updates do not depend on `Thing`'s background enrichment. This lets an enchantment move from inflict-on-target to on-equip without preserving the old target effect or losing regenerated mechanical stats.
- The shared API scope passed into `api.js` includes both `parseThingSeparateResponse` and `separateThingByPrompt`, so `POST /api/things/:id/separate` can invoke prompt-backed separation directly instead of failing on a missing helper reference.
  - Base-context `<currentLocation><npcs>` entries exclude party members because those actors are listed separately under `<playerParty>`, and include raw `resistances`, `vulnerabilities`, and `aiNotes` strings so combat/attack/NPC-turn prompts can reason about effectiveness and action triggers from NPC profiles.
  - NPC generation prompts now include a shared contemplation/self-correction step sequence (free-text steps followed by final XML output) to improve NPC planning quality before parsing.
  - NPC XML parsing for location/region/single generation now captures `resistances`, `vulnerabilities`, `<aiNotes>`, `<startingNeeds>`, and `<startingHealthPercentage>` entries. Generated `<startingLevel>` values are treated as percentages of the bar max (optional trailing `%` is ignored), then converted into concrete starting values before `Player` creation. `<startingHealthPercentage>` also accepts an optional trailing `%`; `deceased` is treated specially and finalizes the generated NPC as a persistent corpse (`persistWhenDead=true`) after inventory/equip generation so dropped items stay consistent. Generated NPCs also persist a dedicated `needBarApplicability` map, so bars explicitly marked inapplicable stay disabled for that actor across save/load instead of silently reappearing from default definitions.
  - NPC generation now includes alias generation (`npc_alias_assignments`) via standard `LLMClient.logPrompt` logging. Alias prompts run alongside ability/inventory generation paths, aliases are persisted on `Player` objects, `ensureNpcByName` resolves in order (full names, party aliases, same-location aliases, global aliases), and k-gram normalization excludes both NPC name and alias tokens. When parsed/generated NPC data collides with an NPC already in the same target location, the duplicate candidate is skipped instead of being renamed; single-NPC/event generation reuses the existing same-location NPC, and region generation preserves any reused same-location important NPCs instead of deleting them during rebuild.
  - Base prompt context now includes a `<factions>` block for LLM generation; location/region generators and NPC generators expect faction names (or "None"), and server-side resolution now matches case-insensitively with punctuation stripped before mapping names to ids.
  - If a generated faction name still does not match, faction resolution logs a server warning with the unmatched name and falls back to no faction instead of aborting generation.
  - Region-entry stub expansion now applies defensive controlling-faction reconciliation: unknown stub/region faction ids are ignored with warnings, generated faction mismatches do not abort expansion, and existing saved region faction is preferred when conflicts occur.
  - Region-entry finalization now rewires any remaining inbound exits that still target the consumed stub to point at the finalized entrance location (dropping accidental entrance self-loops and rebuilding reverse links for bidirectional exits), removes stale `locationIds` and `entranceLocationId` references to the consumed stub across all regions (repairing non-target region entrances to valid fallbacks when available), remaps region vehicle `vehicleInfo.vehicleExitId` from consumed-stub exit ids onto the replacement entrance exits, then removes the consumed stub from both `gameLocations` and `Location` static indexes, and asserts post-finalization integrity (no remaining region references/exits/player location pointing at the removed stub). During region-entry stub expansion, generated connected-region definitions that point back to the source region are now filtered before stub creation so expansion does not create a second return path to the origin region. Detailed exit-link diagnostics for this path are available only when starting the server with `--test=region-exits` (legacy alias: `--debug-region-exits`), and vehicle icon/exit-direction diagnostics are available via `--debug-vehicles` (or `--test=vehicles`).
  - Region generation/stub expansion now parse `<location><vehicles>` blocks. Region stub location parsing reads only direct `<region><locations><location>` children, so nested vehicle-destination `<location>` tags are not misparsed as region locations. `large` and `huge` vehicles are promoted into navigable stubs: large vehicles become in-region location stubs, huge vehicles become pending region-entry stubs; repeated route vehicles are deduped by name+size, boarding exits are created with vehicle semantics, parsed `<destinations>` are resolved to location IDs for `vehicleInfo` route metadata when possible, and blank/missing `<icon>` values default to `🚗`. Both region-entry stubs and normal event-created location stubs now persist whether origin-exit creation was intentionally suppressed, so later arrival-time stub expansion does not accidentally create a second plain exit alongside the tracked vehicle exit. Event-created location stubs now also persist an explicit `relativeLevelBase` plus computed stub base level from the origin location, while event-created region-entry stubs stamp a computed stub base level from the current region average, so later expansion keeps relative difficulty anchored to the correct source. Explicit prompt-generated `0`-minute exit travel times are normalized up to `1` minute during region and region-stub parsing, while save hydration leaves missing exit times at `0` so `/fill_exit_travel_times` can target only still-unpopulated exits. Generated `<regionExits><stubRegion><travelTime>` values are now parsed too and applied directly to the created cross-region exits without persisting travel time on the pending stub metadata. If region-entry stub expansion fails during instantiation, the server now rolls back the partially created region/locations, logs a warning, and leaves the pending stub unresolved instead of continuing with a half-built region; standalone region generation still throws after rollback.
  - Location/region name regeneration now filters alternate name candidates through the same banned-location-name and slop-word checks before any candidate can be assigned. The filter also includes single-word active-setting custom slop entries, removes duplicates/original names, and returns only the surviving candidates in regeneration metadata.
  - Faction generation now runs in three prompts: core factions (`faction_generation`), inter-faction relationships for all factions (`faction_relationship_generation`), then reputation tiers for all factions (`faction_reputation_generation`). Each stage retries up to 3 times on stage failure (render/prompt/LLM/parse/validation errors). Missing/invalid relation entries are ignored or normalized, unspecified inter-faction relations default to neutral with a default note, missing faction assets default to an empty list (with warnings), surplus generated factions beyond `config.factions.count` are accepted (only too-few factions fail new-game setup), and reputation-tier output must include tiers for every generated faction.
  - Prompt templates now also include `prompts/faction-inbound-relationships-generator.xml.njk` for one-pass generation of existing factions' dispositions toward a newly created faction.
  - NPC progression now uses formula-driven point pools for both skills and attributes (`formulas.character_creation.*`). NPC prompts provide priority lists (not exact numbers), and the server spends pool deltas in one pass, including multi-level gains from a single level-up event. Creation and level-up flows log computed attribute/skill budgets for debugging.
  - NPC post-generation skill/ability prompts now run as fresh base-context prompts instead of continuation turns: they receive the raw generated NPC XML as `generated_npc_results`, receive raw generated location/region XML as `generatedRegionOrLocation` when applicable, location/region flows still launch skills and abilities concurrently after NPC generation, and single-NPC generation now does the same.
  - NPC level-up ability generation now supports required-addition mode with up to 3 attempts and explicit excluded ability-name guidance in the prompt payload, so regeneration paths (like NPC ability respec) fail loudly instead of silently producing zero unique additions.
  - Player level-up ability assignment is now a player-only draft flow instead of automatic assignment: missing levels are detected from level 1..current level using `player_abilities_per_level`, pending state can be resolved without generation, option pools are generated on explicit ability-selection fetch (`generateOptions=true`) so the modal can open first, generated options are persisted on the `Player` record, and gameplay stays blocked until pending levels are submitted in order.
  - `generateItemsByNames` now accepts generation `options` and only parses as inventory when an owner is provided, so event/location generation can preserve `scenery` outputs (including `treatAsScenery` / `treatAsResource` flows) instead of coercing everything to `item`.

Character portrait generation is de-duped both during pre-job prompt generation and after the image job exists, so duplicate requests join the existing work instead of queuing multiple renders. ComfyUI initialization logs a warning and continues if the server is unavailable while keeping the client enabled for later startup. - api.js
  - Exports `registerApiRoutes(scope)`; validates scope (must include Express app plus numerous helpers) and optionally installs an axios AI debug interceptor. Uses `with (scope)` to share state. Utility helpers inside include stream emitters for realtime (`realtimeHub`), autosave pruning, XML helpers for crafting/random events, and deletion helpers for NPCs/things.
  - `/api/craft` covers craft/process/salvage/harvest actions. It resolves the craft plausibility/success-degree `ActionResolution` into the same visible prompt-excluded `check-results` chat entry shape used by skill checks, then keeps the pre-action autosave checkpoint and also attempts a standard autosave after successful state mutation, so completed inventory/scenery changes, applied time, harvest history, vehicle arrivals, and NPC sighting updates are captured before the response is sent. `/api/locations/:id/modify` uses the same synthesized `check-results` path for its location-modification success degree.
  - Chat tool-call definitions and execution runtime are provided by `chat_tool_calls.js`; `api.js` wires it with scope accessors and uses the exported `CHAT_TOOL_DEFINITIONS` plus runtime helpers. Runtime dependency guards accept class/function exports as well as objects. Per-round `*_tool_loop_round` logs now include a dedicated `TOOL CALLS` section listing each invoked tool name, call id, and raw JSON arguments for that round. Tool execution errors are logged under a `*_tool_call_error` prompt prefix and returned to the model as `<toolError>` tool messages instead of aborting the prompt. The information-tool set includes `getFullScene({ sceneNumber })`, which resolves stored `Scene N` entries listed inside `<olderStoryHistory>` through the shared scene-summary index and returns delineated action-input/prose entries for that scene.
- New-game/load lifecycle also includes calendar generation support: `/api/new-game` requests an LLM-generated calendar definition using setting context (`calendar_generation` metadata label), and the prompt explicitly requires Gregorian output when the setting is reasonably Earth-based while also requesting season descriptions, seasonal `timeDescriptions`, and 10 holidays (with descriptions). Load regenerates one only when a save is missing `calendarDefinition`; both paths fall back to Gregorian on failure.
- `/api/new-game` and `/api/load` both call the shared config/defs reload path in strict need-bar sentence-validation mode, so missing need-bar prompt sentences abort those lifecycle actions before game state is cleared/hydrated. Live `/reload_config` and per-game `/api/game-config-override` reloads downgrade the same validation to warnings so a running session is not blown away mid-development.
- `/api/new-game` now resolves player startup ability-selection state before the opening-scene step, marks the runtime as loaded at that point, and defers the base-context intro prompt (`game_intro`) when the player still has pending level picks; the final `/api/player/ability-selection/submit` that clears those pending picks consumes the deferred-intro metadata, generates the visible `game-intro` assistant entry, and then the shared intro helper attempts the standard autosave flow. Intro-generation failures and intro-triggered autosave failures still log warnings instead of failing the overall flow.
- `api.js` exposes `Globals.generateGameIntro` so slash commands can trigger the same intro prompt on demand.
- Game intro prose now runs through the slop-removal pipeline (`applySlopRemoval`) before the `game-intro` chat entry is stored, emits a linked `slop-remover` attachment entry when slop checks run (including slop words, configured regex names, and n-grams), and then attempts the standard autosave flow for both new-game and slash-command intro generation.
- `/api/chat` is the main turn handler: applies status effect need deltas, optional travel metadata, autosave (autosaves skip the newest chat entry if it matches the latest autosave entry id), accepts empty player-action continuations by skipping plausibility before rendering the player-action prompt, builds prompts via `prepareBasePromptContext` + `promptEnv`, and then chooses a prompt-mode-specific completion path: regular prompts, including `player_action` and `npc_action`, are restricted to information tools plus the `resolveAttack` exception and skill-check tools (`moreInfo`, `getHistory`, `getFullScene`, `listLocationEntities`, `resolveAttack`, `resolveSkillCheck`, `resolveOpposedSkillCheck`, `locateNpcs`, and `locateThings`); `resolveAttack` computes the attack roll (attack rolls add level bonuses: ceil(attacker/2) to attack skill + damage attribute; floor(defender/2) to evade/deflect), applies defender damage, returns only damage and remaining-health percentages (or `miss`), and emits one attack summary per tool call; `resolveSkillCheck` and `resolveOpposedSkillCheck` call the same `resolveActionOutcome` mechanics as the old plausibility path and emit skill-check/plausibility metadata, defaulting omitted `actor` to the acting NPC for `npc_action` prompts and to the current player otherwise; attack/skill-check tool results are cached within each prose prompt round by round id plus actor/attacker, target/opponent when applicable, weapon/ability/attribute when applicable, and skill so same-prompt relitigation returns the original result without re-rolling or re-applying damage, while appending a warning note to the cached tool content the model sees; a new prose prompt receives a fresh cache; skill and attack tool results are mirrored into one visible prompt-excluded `check-results` chat entry per prompt and updated through `chat_history_updated` as checks start and complete; when `debug_tool_calls` is true, prose-prompt tool calls are also mirrored into one separate visible prompt-excluded `tool-call-debug` chat entry with structured records, cache-hit markers, rendered as collapsible per-tool-call sub-boxes, and updated through `chat_history_updated` as calls start and complete. If `use_legacy_prompt_checks` is true, regular prose prompts omit `resolveAttack`/skill-check tools and re-enable the older separate prompt flow: player actions run `attack_precheck`/`attack_check` plus automatic plausibility before prose, and NPC turns run their action-plan plausibility plus legacy attack check before NPC prose. `@`/`@@`/`@@@` generic prompts keep the generic prompt flow with the full world-mutation tool set including character/item teleport, thing creation via `thing-generator-single` including container-flag seeds, whole-stack thing alteration via `thing-alter`, NPC alteration via `alter_npc`, location alteration via `alter_location`, `createRegionStub`, `createLocationStub`, and `createExit(...)`; `\` prompts render through `prompts/generic-prompt-nocontext.xml.njk`, persist with `excludeFromBaseContextHistory`, and disable chat tools entirely. For world-mutation tools, recoverable errors are returned to the model as structured `<toolError>` payloads with disambiguation candidates instead of aborting the chat call. The route logs prompts and streams status back to the client. Status effects now track `appliedAt` world time and tick per elapsed in-game minute (with finite durations decremented in minute units, capped to remaining duration). It records chat history, injects NPC/location memories, may trigger `Events.runEventChecks`, advances canonical world time from `time_passed` event outputs (leading reasoning ignored; final duration parsed as minutes via shared duration parsing), includes `worldTime` payloads in chat responses/history (`timeMinutes`, season, and holiday descriptions when available), and schedules hidden asynchronous post-turn prompts:
  - Split `<travelProse>` parsing accepts `<playerDestination><travelTime>...</travelTime></playerDestination>`, normalizes it through the generated-exit duration path, and passes it only when resolving the player destination creates a new origin exit. Newly created return exits copy the same duration, while existing exits are deliberately not retimed by this prompt field.

Base-context also includes a `<lastSeenNpcs>` block for NPCs with persisted last-seen data who are not currently present with the player, explicitly labeling it as memory rather than current scene presence.   - Player/tool plausibility skill-check parsing supports both unopposed (`<difficulty><unopposedCheck><difficultyLevel>...`) and opposed (`<difficulty><opposedCheck>...`) XML structures; opposed checks resolve the referenced opponent actor (name/loose-name/alias) and run contested d20 outcomes.   - Repetition-buster player-action/creative-action XML may return `<rejected>...</rejected>` instead of prose. The parser accepts rejection reasons from either text content or XML comments; `/api/chat` displays that reason like a rejected plausibility check, stores a linked plausibility attachment, marks the saved user entry and visible rejection message with `metadata.excludeFromBaseContextHistory`, and returns immediately before normal response storage, slop removal, event checks, random events, or NPC turns.
  - Player-action prose can include `<hidden>...</hidden>` note blocks.   - On player arrival, `/api/chat`, `/api/player/move`, and player teleport perform a blocking `while-you-were-away` base-context prompt before the destination scene is serialized. `whileYouWereAwayNpcs` now includes all current-location reunion candidates with valid last-seen data, and `needBarDefinitions` includes each bar's `while_you_were_away_prompt_notes`, but the prompt still only runs when at least one candidate has been away at least `while_you_were_away_threshold_minutes` (default `240`).
