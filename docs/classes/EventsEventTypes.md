# Events Event Types

## Purpose
Reference for the non-dummy event types recognized around `Events.js`. This excludes `dummy_event` and `_instructions_` prompt entries.

Most rows come from the numbered grouped event checks in `EVENT_PROMPT_ORDER`. A few rows are pipeline-only: they are parsed and handled by `Events.js`, but are injected by another prompt or helper rather than by an active numbered event-check question.

For the planned single-prompt XML event pipeline, see [EventsXmlEventSchema.md](EventsXmlEventSchema.md).

## Event Type Table

| Event type | Primary source | Parsed payload | Meaning and application |
| --- | --- | --- | --- |
| `new_exit_discovered` | Numbered location event checks and XML `newExitDiscovered` | Destination name, destination kind (`location`/`region`), optional vehicle type, description, optional travel time, optional exit/source location (XML `origin.locationName`), optional exit/source region (XML `origin.regionName`), optional destination region, optional destination location name | A new route, door, path, portal, or vehicle route was discovered or created. The handler creates or wires the destination/exit without moving the player. New travel times are parsed as minutes and attached to the created exit wiring. Explicit source locations let the event add exits from locations other than the active one; if the named source location is missing but its source region resolves, the handler creates that source as a region-local stub without adding a current-location exit. Destination-region hints are used when resolving or creating location destinations. If XML marks the destination as a region but includes a destination location, the parsed region name remains the handler target and the destination location is preserved for summary text. |
| `move_new_location` | Numbered location event checks | Difference text, new destination name, kind (`location`, `region`, or `sublocation`), optional vehicle type, description | The player/party moved to a destination that does not already exist in the world model. The handler creates/discovers the destination and moves the player there unless movement is suppressed for split travel-prose checks. `sublocation` is treated as a real location when applied. |
| `move_location` | Numbered location event checks | Existing destination location name | The player/party moved to an already-known location. The handler resolves the final destination and moves the player there, with single-player-move turn guards in `movePlayerToDestination(...)`. |
| `alter_location` | Numbered location event checks | Current location name, new location name, change description | The current location's visual/environmental description changed without the player simply moving away. The handler invokes the location alteration prompt, applies the returned location XML, clears location imagery when needed, and records alteration summaries. |
| `currency` | Numbered item event checks | Signed integer delta | The player gained or lost currency. The aggregator sums all currency answers and the handler modifies the player currency by the resulting delta. |
| `item_inflict` | Numbered item event checks | Item name, target name, prompt-provided status text | An item was used in a way that might activate its target effect. The handler ignores the prompt's status text and applies the item's configured `causeStatusEffectOnTarget` to the target when available. It also emits item-triggered status-change summary data. |
| `item_ingest` | Numbered item event checks | Item name, target/consumer name | Someone ate, drank, or otherwise ingested an item. The handler infers the applied effect from the ingested item's configured target effect and suppresses duplicate same-pair `item_inflict` applications. |
| `item_to_npc` | Numbered item event checks | Original item/scenery name, new NPC/entity name, short description | An inanimate object became an independent moving entity. The handler ensures/generates the NPC, places it in the current location, tracks the new character, and can remove or detach the source thing as appropriate. |
| `alter_item` | Numbered item event checks | Original item name, quantity or `all`, new item name, change description | An item or scenery object was permanently changed. The handler resolves the source thing, splits stacks for partial quantities, runs the thing alteration prompt, applies the updated item/scenery data, and refreshes images/stats/metadata. |
| `consume_item` | Numbered item event checks | Item name, quantity, consumption reason | An item or scenery object was fully used up, destroyed, eaten, or assembled into something else. The handler removes or decrements exact quantities, including stack-aware consumption. |
| `transfer_item` | Numbered item event checks | Giver name, item name, quantity, receiver name | An item was handed, traded, or given from one actor to another. The handler moves exact stack quantities between inventories. |
| `pick_up_item` | Numbered item event checks | Actor name, item name, quantity | A character obtained carryable tangible items by a method other than harvesting/gathering. The handler creates/ensures the item if needed and places exact quantities in the actor's inventory. |
| `harvest_gather` | Numbered item event checks | Harvester name, item name, quantity, optional source name | A character gathered resources from a harvestable source, pile, plant, vein, machine, etc. The handler creates/ensures the harvested item, places it in inventory, and records successful harvest history on the source node when available. |
| `item_appear` | Numbered item event checks | Item name, quantity, description | New carryable items appeared in the scene for the first time. The handler generates/creates item things in the current location, preserving quantities. |
| `drop_item` | Numbered item event checks | Character name, item name, quantity | A character dropped, placed, or set down inventory items into the scene. The handler moves exact stack quantities from inventory to the current location. |
| `scenery_appear` | Numbered item event checks | Scenery name, description | New non-carryable scenery appeared in the scene for the first time. The handler generates/creates scenery things in the current location. |
| `harvestable_resource_appear` | Numbered item event checks | Resource/scenery name, description | A new harvestable or gatherable resource appeared. The handler generates/creates scenery marked for resource harvesting where supported. |
| `attack_damage` | Numbered NPC event checks | Attacker name, target name | A physical attack happened. This event is tracked and summarized, but the handler does not directly apply damage; actual attack resolution happens elsewhere before/around narrative generation. |
| `alter_npc` | Numbered NPC event checks | Character name, alteration category, change description | An animate entity was permanently physically or meaningfully changed by something other than normal attack damage. The handler runs the NPC alteration prompt, applies returned NPC XML, and tracks altered character summaries. |
| `status_effect_change` | Numbered NPC event checks plus synthesized item-triggered entries | Entity name, status effect name/detail, `gained`/`lost`, optional level | A temporary status effect was gained or lost. The handler generates mechanical status-effect stats for gained effects when possible, applies or removes the effect on the actor, and de-duplicates gains already applied by item-triggered events. |
| `npc_arrival_departure` | Numbered NPC event checks | NPC/entity name, action (`arrived`/`left`), optional destination region/location | An animate entity arrived in or left the scene. The handler ensures arriving NPCs, places them in the current location, or moves departing NPCs to the resolved destination. Current-player events and accidental party-member arrivals remain suppressed, but an explicit party-member `left` departure removes that NPC from the party before moving them to the resolved destination. |
| `npc_first_appearance` | Numbered NPC event checks | Present entity names | A physically present actor was mentioned but was not already known/listed. It is converted into arrival-style entries during parsing so the normal NPC arrival handling can ensure and place the actor. |
| `party_change` | Numbered NPC event checks | NPC name, `joined`/`left` | A present NPC began accompanying the player or left the party without a separate destination move. The handler adds/removes party membership on the current player. |
| `environmental_status_damage` | Numbered NPC event checks | Actor name, `damage`/`healing`, severity, reason | An actor took environmental/status damage or healing. When environmental effects are allowed, the handler converts severity into a health delta, applies it, and records environmental damage/healing summary data. |
| `heal_recover` | Numbered NPC event checks | Character name, magnitude (`small`/`medium`/`large`/`all`) or healer/recipient/effect, reason | A character recovered health. The handler estimates healing amount from magnitude and actor stats, then applies health recovery. |
| `hostile_to_friendly` | Numbered NPC event checks | NPC name, previous disposition, new disposition, reason | A hostile/unfriendly NPC became neutral, friendly, or allied. The handler clears hostility and raises all configured disposition types to at least neutral. |
| `in_combat` | Numbered misc event checks | Boolean-ish yes/no | The player is currently in physical combat. The handler stores the normalized combat flag in the outcome context for consumers. |
| `received_quest` | Numbered misc event checks | Optional giver, quest/task summary | The player became aware of a quest/task. The handler runs quest generation, asks for confirmation through the quest confirmation manager, and records awarded quests when accepted. |
| `death_incapacitation` | Numbered misc event checks | Actor name, `dead` or `incapacitated` | An entity died or became incapacitated. The handler zeroes health, marks NPCs dead for `dead`, and applies `Deceased` or `Incapacitated` status effects. Duplicate dead outcomes for already-dead NPCs are skipped. |
| `defeated_enemy` | Numbered misc event checks | Enemy name(s) | The player defeated an enemy. The handler awards XP based on enemy/location level and player level, and tracks defeated enemy names. |
| `experience_check` | Numbered misc event checks | Raw XP amount, reason | The player did something that warrants experience aside from defeating enemies. The handler scales the prompt amount by location/player level and awards XP unless an enemy-defeat award already happened this turn. |
| `faction_reputation_change` | Numbered misc event checks | Faction name, increase/decrease magnitude (`a little`/`a lot`), reason | The player's faction standing changed. The parser maps `a little` to `1` and `a lot` to `4`, signed by increase/decrease. The handler applies the change only when a witness from that faction is present with the player, in the party, or at the location. |
| `time_passed` | Numbered misc event checks | Reasoning/breakdown and final duration | The non-travel turn consumed in-world time. The parser reads the final arrow-delimited duration segment and converts it to minutes. The handler advances canonical world time unless suppressed, immediately applies time-based need/status ticks, and `0` still advances by one minute to avoid static clocks. Successful travel suppresses this event so route/exit time stays authoritative. |
| `triggered_abilities` | Numbered misc event checks | Character name, ability name | Prompt-only at present. `Events.js` records raw entries and uses them for NPC name ensuring, but there is no parser/handler that applies triggered ability effects. |
| `needbar_change` | Dedicated need-bar prompt injected into event outcomes | Character name, need bar name, direction, magnitude, optional reason | A need bar changed. The active source is the parallel `need-bars.njk` prompt, not the commented old numbered event question. The handler calls `applyNeedBarChange(...)` on the actor and records `needBarChanges` for summaries/client payloads. |
| `disposition_check` | Pipeline-supported legacy/structured event, not an active numbered prompt question | NPC name, before feeling, after feeling, reason | A significant NPC disposition toward the current player changed. The handler resolves direction from before/after text, applies one typical disposition step to the default/platonic disposition type, and records `dispositionChanges`. Current async memory/disposition prompting in `api.js` uses its own XML flow, but this Events handler remains available for structured event outcomes. |
| `completed_quest_objective` | Quest-check merge pipeline, not an active numbered event question | Quest index/objective index, later normalized to quest/objective IDs and descriptions | A quest objective was completed. `runQuestChecks(...)` and `mergeQuestOutcomesIntoStructured(...)` inject normalized objective completions into event outcomes. The handler marks objectives complete, applies quest-completion rewards when appropriate, and preserves prompt-supplied completion reasons. |

## Notes
- `dummy_event` entries are deliberately excluded because they are scaffolding questions used to steer later answers, not applied outcomes.
- `_instructions_` entries are excluded because they are prompt instructions, not events.
- The application order starts from `EVENT_PROMPT_ORDER_FLAT`, then appends any parsed keys not present there. This lets pipeline-only types such as `needbar_change` and `completed_quest_objective` still run. Because `needbar_change` is appended after `time_passed`, event-authored need-bar changes resolve after elapsed-time ticks.
- Some event types are suppressed by context flags: `suppressMoveEvents` skips `move_location`/`move_new_location`; `omit_npc_generation` skips some NPC generation/alteration; `omit_item_generation` skips item/scenery generation and alteration.
